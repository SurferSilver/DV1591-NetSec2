\section{Introduction}
As of late 2025, the Google Threat Intelligence Group (GTIG) has observed a notable shift in how threat actors utilise artificial intelligence. Rather than merely using AI as a coding assistant, adversaries are now embedding Large Language Model (LLM) capabilities directly into their attack workflows. This development includes malware families that employ ``Just-in-Time'' AI, which generates malicious scripts and obfuscates code dynamically at runtime instead of relying on static, pre-written logic.

\section{Bypassing Guardrails via Social Engineering}
Threat actors have developed techniques to circumvent the safety guardrails implemented in commercial AI systems. These techniques primarily rely on social engineering within the prompts themselves.

One common method involves Capture-the-Flag (CTF) pretexts, where actors pose as security researchers or students participating in cybersecurity competitions. This framing can persuade models to provide exploitation-related information that would otherwise be blocked. Another approach uses academic lures; for example, Iranian state-sponsored actors such as TEMP.Zagros have been observed framing their queries as university students working on final projects or academic articles, thereby soliciting assistance with malware development.

\section{Case Study 1: PROMPTSTEAL (LAMEHUG)}

\subsection{Target Characteristics and Delivery}
PROMPTSTEAL, designated \texttt{LAMEHUG} by CERT-UA, is a Python-based data mining tool designed for reconnaissance and document exfiltration. The campaign primarily targets Ukrainian government authorities and critical infrastructure organisations.

The malware uses multiple delivery methods to increase the likelihood of successful compromise. The phishing variant is distributed via emails disguised as official ministry correspondence, with the primary lure being an archive named \texttt{Appendix.pdf.zip} containing a \texttt{.pif} executable. Alternatively, the deceptive application variant masquerades as an AI image generation tool called \texttt{AI\_generator\_uncensored\_Canvas\_PRO\_v0.9.exe}. This variant displays a functional user interface while covertly querying the Hugging Face API in the background.

\subsection{Attribution}
The campaign has been attributed with high confidence to \textbf{APT28}, also known as FROZENLAKE or Fancy Bear, a threat group linked to Russian government operations. APT28 has integrated the \texttt{Qwen2.5-Coder-32B-Instruct} model into their operational workflow, using stolen API tokens to query Hugging Face. This approach enables dynamic command generation, providing greater operational flexibility compared to traditional hard-coded malware logic.

    \caption{Attribution to APT28 (Fancy Bear).}

\subsection{Technical Analysis}
The core functionality of PROMPTSTEAL centres on ``Just-in-Time'' command generation, where static malicious scripts are replaced with dynamically generated commands from an LLM. Once the malware establishes a connection to the Hugging Face API using stolen tokens, the attack proceeds through two phases.

\textbf{Phase 1: Environment Reconnaissance} \\
The malware first prompts the LLM (\techterm{Qwen2.5-Coder-32B-Instruct}) to interpret the host environment and generate optimized Windows shell commands for system discovery. It specifically requests the output in a clean, one-line format without markdown for direct execution via the Python \techterm{subprocess} module:
\newpage

\begin{quote}
    \textit{"Make a list of commands to create folder C:\textbackslash Programdata\textbackslash info and to gather computer information, hardware information, process and services information, networks information, AD domain information, to execute in one line and add each result to text file c:\textbackslash Programdata\textbackslash info\textbackslash info.txt. Return only commands, without markdown."}
\end{quote}

\textbf{Phase 2: Targeted Document Mining and Exfiltration} \\
Once the system metadata is staged in \techterm{C:\textbackslash ProgramData\textbackslash info\textbackslash info.txt}, the malware issues a second query targeting sensitive user data:

\begin{quote}
    \textit{"Make a list of commands to copy recursively different office and pdf/txt documents in user Documents, Downloads and Desktop folders to a folder c:\textbackslash Programdata\textbackslash info\textbackslash to execute in one line. Return only command, without markdown."}
\end{quote}

After the files are staged, the malware initiates the exfiltration phase to transmit the stolen data to adversary-controlled infrastructure. Forensic analysis of the LAMEHUG source code reveals that the malware utilizes two distinct protocols for exfiltration, depending on the variant:
\begin{itemize}
    \item \textbf{SFTP (Paramiko)}: One variant uses the \techterm{Paramiko} library to establish a secure connection to the hard-coded IP address \techterm{144.126.202.227} on port 22 to upload the staged archive.
    \item \textbf{HTTP POST}: Other variants utilize the Python \techterm{requests} library to send files via \techterm{POST} requests to PHP-based endpoints, such as \techterm{stayathomeclasses.com/slpw/up.php}.
\end{itemize}

\subsection{Indicators of Compromise (IoC)}
Defenders can detect PROMPTSTEAL activity by monitoring for the following technical artifacts:
\begin{itemize}
    \item \textbf{File System}: Creation of the staging directory \texttt{C:\textbackslash ProgramData\textbackslash info} and the presence of compiled Python executables with \texttt{.pif} or \texttt{.exe} extensions.
    \item \textbf{Network}: Outbound API requests to \texttt{router.huggingface.co} or similar LLM hosting platforms.
    \item \textbf{Exfiltration}: SFTP or HTTP POST traffic directed to adversary-controlled infrastructure, such as the IP address \texttt{144.126.202.227}.
\end{itemize}

\subsection{Security Controls and Mitigations}
To effectively detect and block PROMPTSTEAL, organizations should implement the following controls:
\begin{itemize}
    \item \textbf{API Egress Filtering}: Implement strict monitoring or blocking of traffic to AI model APIs (e.g., Hugging Face) from workstations without a verified business need.
    \item \textbf{Behavioral Detection}: Deploy EDR rules to identify the "blind" execution of complex shell commands originating from Python script interpreters.
    \item \textbf{Credential Hygiene}: Enforce policies to prevent the storage of API tokens in cleartext, as these are often harvested by malware to maintain AI connectivity.
\end{itemize}

\section{Case Study 2: QuietVault}

\subsection{Overview and Targets}

QUIETVAULT is a JavaScript-based credential stealer first identified in 2025. The malware targets developer workstations running macOS or Linux, where locally hosted AI tools are increasingly common. Its primary objective is harvesting authentication credentials, particularly GitHub and NPM tokens, as well as cryptocurrency wallet data. The malware identifies local LLM installations such as OLLAMA and uses them to search for various digital wallets, including MetaMask, Electrum, Ledger, Trezor, Exodus, Trust Wallet, Phantom, and Solflare.

\subsection{Attribution}

No attribution to a specific threat actor has been established for QUIETVAULT. Google Threat Intelligence, which documented the malware, noted that similar variants likely exist on other platforms. The focus on cryptocurrency theft and credential harvesting suggests financially motivated cybercrime rather than state-sponsored activity.

\subsection{Technical Analysis}

QUIETVAULT exploits locally installed AI tools to perform reconnaissance. Upon execution, the malware enumerates the system for local LLM installations, focusing on OLLAMA.

When a local LLM is found, QUIETVAULT injects a crafted prompt instructing the model to analyse filesystem contents and identify files containing cryptocurrency wallet data, passwords, and authentication tokens. The prompt specifies that the search should avoid privileged system directories, ensuring the reconnaissance stays within user-accessible paths and avoids triggering privilege escalation alerts.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.8\textwidth]{Images/QUIETVAULT.png}
    \caption{Example of a QUIETVAULT prompt injected into a local LLM to identify sensitive files.}
\end{figure}

This technique offers several advantages. The reconnaissance is performed by legitimate software already on the system, reducing the malware's behavioural footprint. The LLM can identify sensitive files based on content patterns and naming conventions rather than hardcoded paths. Furthermore, since the analysis occurs locally, no network traffic is generated until the exfiltration phase.


\subsection{Indicators of Compromise (IoC)}
Detection of QUIETVAULT requires focusing on the behavioral "bridge" between traditional script execution and local AI interaction. Because the AI reasoning occurs locally, forensic analysis should prioritize internal process telemetry and staging artifacts:

\begin{itemize}
    \item \textbf{Anomalous Process Lineage}: Detection of a JavaScript/Node.js or shell process spawning a local AI CLI (e.g., \techterm{ollama}, \techterm{gemini-cli}) as a child process. This is a primary indicator of "living-off-the-land" AI abuse, as standard developer workflows rarely involve automated script-to-AI-CLI pipes for filesystem enumeration.
    \item \textbf{Local Prompt Artifacts}: Presence of reconnaissance-heavy strings in local AI tool history files or logs. Investigators should monitor for prompts requesting the identification of "wallets," "private keys," or ".env" files, as these natural-language queries are stored locally by many CLI interfaces.
    \item \textbf{Staging and Encoding Artifacts}: The appearance of large, Base64-encoded strings or temporary archive files in hidden directories (e.g., \techterm{/tmp} or \techterm{~/.local/share/temp}) prior to exfiltration. These serve as the "staging area" where harvested data is prepared before being sent to GitHub.
    \item \textbf{High-Frequency Metadata Access}: A rapid, systematic update of "access time" (atime) metadata across a wide range of sensitive configuration directories (\techterm{~/.ssh}, \techterm{~/.aws}, \techterm{~/.config}) within a very short window, indicating an automated search rather than manual developer activity.
    \item \textbf{GitHub API Anomaly}: Detection of a \techterm{POST /user/repos} request to the GitHub API. In a corporate or professional setting, creating an entirely new, public repository via an automated script from a local workstation is a high-confidence signal of exfiltration, especially if the account has no history of such actions.
\end{itemize}


\subsection{Security Controls}
To mitigate the risk of such AI-augmented stealers, the following security controls are recommended:
\begin{itemize}
    \item \textbf{Filesystem Hygiene}: It is important that users and developers never store cryptocurrency wallets, private keys, or plain-text authentication tokens directly on the local filesystem. These assets should be managed via hardware security modules (HSMs), hardware wallets, or encrypted vault solutions.
    \item \textbf{Endpoint Hardening}: Implement strict filesystem access control lists (ACLs) to prevent local AI binaries from reading sensitive configuration directories or wallet keystores without explicit user authorization.
    \item \textbf{Credential Management}: Enforce the use of short-lived, fine-grained personal access tokens (PATs) for GitHub and NPM, and monitor for the creation of unauthorized public repositories within the organization's environment.
\end{itemize}

\section{Case Study 3: PROMPTFLUX}

\subsection{Overview}

PROMPTFLUX demonstrates the use of generative AI for automated code polymorphism. This \textbf{experimental} dropper, implemented in VBScript, queries the Google Gemini API to modify its own source code, producing functionally equivalent but syntactically distinct variants that can evade signature-based detection.

The key component is the ``StartThinkingRobot'' function, which interfaces with the Gemini API to request code modifications. Unlike traditional polymorphic malware that uses predefined mutation rules, PROMPTFLUX uses the LLM's understanding of code semantics to generate varied transformations that preserve functionality while changing surface-level indicators.

\subsection{Implications for Detection}

Google has documented PROMPTFLUX variants that rewrite their entire source code hourly. This mutation frequency exceeds what traditional polymorphic engines can achieve while maintaining functional code, presenting challenges for detection.

Signature-based detection relies on identifying static patterns in malicious code. When malware regenerates itself continuously through AI-guided transformations, such stable patterns do not exist. Each malware instance becomes effectively unique, necessitating a shift toward behavioural detection methods.

If PROMPTFLUX-style techniques become widespread, the economics of malware detection shift toward attackers. Generating undetected variants becomes inexpensive, while defenders must bear the computational cost of behavioural analysis for each potential threat.

\section{Conclusion}
The technical analysis of PROMPTSTEAL, QUIETVAULT, and PROMPTFLUX underscores a fundamental shift in the threat landscape where AI is no longer just a tool for preparation but a core component of the execution phase. This transition to "Just-in-Time" AI-driven malware means that defensive strategies must move beyond static file analysis toward monitoring the behavioral interactions between standard system processes and AI interfaces. As threat actors continue to weaponize both cloud-based and local LLMs, the resilience of organizational security will depend on dynamic detection and the strict enforcement of API and filesystem hygiene to disrupt these adaptive attack cycles.

Consequently, the identification of behavioral anomalies has become the primary requirement for effective modern detection strategies. By focusing on the behavioral bridge between standard system processes and AI interfaces, defenders can identify high-confidence signals that traditional signatures fail to flag, such as anomalous process lineages where a script interpreter unexpectedly spawns an AI command-line tool. Furthermore, monitoring for behavioral patterns like rapid, systematic filesystem reconnaissance or unauthorized interaction with local LLM binaries allows for the detection of "living-off-the-land" AI abuse even when external network traffic is minimal. Ultimately, as threat actors weaponize both cloud-based and host-resident AI infrastructure, security controls must evolve into a dynamic framework that prioritizes the monitoring of operational behavior to mitigate these highly adaptive and reasoning-driven threats.
